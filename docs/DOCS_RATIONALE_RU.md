# DOCS_RATIONALE (RU): почему структура документации именно такая

Актуально на: 2026-02-04 12:00 UTC

## Коротко
Мы держим документацию компактной и предсказуемой:
- один вход (`docs/INDEX_RU.md`)
- один “протокол работы” (`docs/WORKFLOW_CONTRACT_RU.md`)
- три “смысловых” документа (PRD/TECH/OPS)
- CHANGELOG для заметных изменений
- ADR для редких необратимых решений

## Принцип
Документация разделена по назначению (идея Diátaxis):
- README/INDEX — быстрый старт и навигация
- OPS — how-to: процедуры “что делать когда”
- TECH — reference: точные контракты/инварианты (MUST/SHOULD)
- PRD — explanation: границы и цели продукта

## Контракты и версии
Публичные контракты (API/инварианты) объявляются в TECH и меняются осознанно.
В SemVer публичный API может существовать строго в документации; поэтому TECH обязателен как контракт. 

## Почему не плодим файлы
Дублирующиеся документы неизбежно начинают противоречить друг другу. Поэтому:
- сначала обновляем существующие документы
- новые файлы добавляем только при появлении нового типа артефакта (INDEX/CONTRACT/ADR/CHANGELOG)
